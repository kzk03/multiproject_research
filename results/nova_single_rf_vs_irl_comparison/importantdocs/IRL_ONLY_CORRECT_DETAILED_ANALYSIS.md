## 5. IRL vs RF: 予測戦略の違い

### 5.1 RF の予測パターン

1. **レビュー数への過度の依存**:

   - 50 件以上のレビューがあると 0.6-0.9 の高確率
   - CI/Bot の大量レビュー（数百〜千件）でも継続と誤予測

2. **継続率の軽視**:

   - 継続率 0-5%でも、レビュー数が多ければ継続予測

3. **後期での極端な低評価**:

   - 9-12m 期間で真の継続者に対し 0.08-0.42 の低確率

4. **少数レビューでの過信**:
   - 3-6 件でも 0.5 超の確率を出力

### 5.2 IRL の予測パターン

1. **バランスの取れた特徴量評価**:

   - レビュー数と継続率を総合的に判断
   - 0.43-0.48 の慎重な確率範囲

2. **継続率の重視**:

   - 継続率が低い（0-5%）→ 0.45 台で離脱予測
   - 継続率が高い（60-70%）→ 0.47-0.48 台で継続予測

3. **CI/Bot パターンの識別**:

   - 大量レビュー + 低継続率 = 離脱パターンと認識

4. **時系列的安定性**:
   - 同一開発者の複数期間で一貫した確率
   - auniyal@redhat.com: 0.4756 → 0.4759（安定）

### 5.3 閾値 0.5 との関係

**IRL の特徴:**

- ほとんどの予測が 0.45-0.48 の範囲
- 閾値 0.5 に対して非常に微妙な判定
- **継続率が高い場合**: 0.47-0.48（継続寄り）
- **継続率が低い場合**: 0.43-0.46（離脱寄り）
- **中間的な場合**: 0.46-0.47

**RF の特徴:**

- 0.08-0.99 と極端に広い範囲
- 0.5 を大きく超える/下回る確信的予測
- しかし、**その確信が誤りであるケースが多数**

## 6. IRL が成功した理由の考察

### 6.1 特徴量の重要性の違い

**IRL が重視している可能性が高い特徴:**

1. **history_rate（継続率）**:

   - 最も信頼できる継続性の指標
   - CI/Bot の識別に有効（低継続率）

2. **history_count と history_rate の組み合わせ**:

   - 大量レビュー × 低継続率 = Bot/CI（離脱）
   - 少量レビュー × 高継続率 = 継続の可能性
   - 中量レビュー × 低継続率 = 離脱

3. **時系列的な一貫性**:
   - 同一開発者の過去期間での行動パターン

**RF が過度に重視している可能性が高い特徴:**

1. **history_count（絶対的なレビュー数）**:

   - 多い = 継続という単純な関連性

2. **その他の活動指標**:
   - レビュー数に相関する他の特徴量

### 6.2 学習パターンの違い

**Random Forest:**

- 決定木の集合による単純な閾値ベース判定
- 「レビュー数 > X → 継続」のような単純ルール
- 特徴量間の複雑な相互作用を捉えきれない
- CI/Bot の「大量レビュー + 低継続率」パターンを学習できない

**Inverse Reinforcement Learning:**

- 開発者の行動から潜在的な「報酬関数」を学習
- 継続/離脱の決定に至る「意図」や「動機」をモデル化
- 複数特徴量の非線形な組み合わせを自然に扱える
- **行動の質（継続率）** と **行動の量（レビュー数）** を区別可能

### 6.3 ケーススタディ

**Case 1: aleksey.stupnikov@gmail.com（離脱）**

- レビュー数: 96 件（中程度）

## 7. IRL の限界と課題

### 7.1 確率範囲の狭さ

- 0.43-0.48 の狭い範囲に集中
- **予測の信頼度が判断しにくい**
- 閾値調整に敏感

### 7.2 例外ケースの存在

- emc.scaleio.ci@emc.com（CI 系だが継続）
- christian.rohmann@inovex.de（継続率 0%だが継続）
- これらのケースでは**文脈情報**が不足

### 7.3 少数サンプルでの不確実性

- レビュー数 3-7 件の開発者では統計的に不安定
- 継続率 66.7%（2/3 件継続）は偶然かもしれない

## 8. 結論と示唆

### 8.1 IRL の優位性

1. **複合的な特徴評価**: レビュー数と継続率のバランス
2. **Bot/CI 系の識別**: 大量レビュー + 低継続率パターンの認識
3. **行動の質の重視**: 単なる量ではなく継続性の評価
4. **時系列的一貫性**: 同一開発者で安定した予測

### 8.2 RF の弱点

1. **レビュー数への過度の依存**
2. **継続率の軽視**
3. **極端な確率出力**（過信）
4. **Bot/CI パターンの誤認識**

### 8.3 今後の改善方向

1. **RF の改善**:

   - 継続率を重視した特徴量エンジニアリング
   - Bot/CI 判定フラグの追加
   - クラス重み付けの調整

2. **IRL の改善**:

   - 確率範囲の拡大（より確信的な予測）
   - 文脈情報の追加（企業、役割など）
   - 少数サンプルでのロバスト性向上

3. **アンサンブル手法**:
   - IRL と RF の予測を組み合わせる
   - IRL が 0.45-0.48 の微妙な範囲の場合、RF の特徴量重要度を参考にする

### 8.4 論文への示唆

1. IRL は**行動の意図を学習**することで、表面的な指標に惑わされない
2. CI/Bot の自動活動と人間の開発者の違いを識別できる
3. 継続率という**行動の質的指標**が予測性能の鍵
4. 少数サンプルでも継続率が高ければ継続を予測可能（ただし要注意）

#### 特徴量まとめ（訓練期間・評価期間で何が使われるか）

この 10 パターンは「訓練期間 → 評価期間」のように見えますが、IRL 側の cross-temporal 実装では **評価の基準日（cutoff/context）は固定（例: 2023-01-01）**で、`eval_period` は「基準日から何ヶ月先を“将来窓”としてラベル化するか（0-3m / 3-6m / …）」を表します。そのため、`history_count/history_rate` は **基準日までの履歴**から算出され、評価窓が変わっても基本的に一定になります。

## 10. 特徴量重要度を踏まえた統合考察（論文向け）

本ケーススタディでは「IRL のみ正解」となる開発者が一定数観察され、特に CI/Bot 系や低い継続率にもかかわらずレビュー数が多い開発者で Random Forest（RF）が継続と誤予測しやすい一方、9-12m の後期パターンでは真の継続者に対して RF が過小評価しやすいことが確認された。ここでは、別途算出した特徴量重要度（RF: Gini importance、IRL: 勾配ベース平均）を参照しながら、なぜこのような「IRL のみ正解」が生じるのかを、モデルの帰納バイアス（何に引っ張られやすいか）として整理する。

重要度の比較から、RF は少数の特徴に強く依存する傾向がある。具体的には、応答速度（約 26.6%）と総レビュー数（約 25.5%）の 2 つだけで重要度の過半を占め、最近の活動頻度（約 16.2%）を加えると上位 3 特徴で約 7 割に達する。一方で、最近の受諾率（約 4.2%）や活動トレンド（約 0.32%）は相対的に低い。この集中構造は、訓練データ内で分岐に寄与しやすい「単純なスカラー量」（平均・総数・頻度）が決定境界の主要因となり、時間的文脈や行動の質の情報が相対的に周辺化されることを意味する。

対照的に、IRL 側の重要度（勾配の絶対値が大きい特徴）では、総レビュー数に加えて協力度、総コミット数、コード品質スコア、強度（ファイル数）といった「協働・貢献の質」や「行動に近い指標」が上位に現れる。また平均活動間隔や活動トレンドには負の勾配が付いており、活動間隔が開くことやトレンドの変化が継続を抑制する方向に働くことが示唆される。ここで注意すべき点として、本節で参照する重要度はあくまで平均的な影響（全体平均）であり、さらに本ファイルで用いている `history_rate`（履歴の受諾率）と、重要度分析で言及される `recent_acceptance_rate`（直近の受諾率）が同一の定義でない可能性がある。それでも、RF と IRL の間で「量（総数・平均）に寄るか、協働行動や活動の一貫性に寄るか」という軸が明確に異なる点は、IRL のみ正解が系統的に発生する理由の説明として有効である。

まず、CI/Bot 系やレビュー総数が非常に大きいにもかかわらず継続率が極めて低い開発者（例: `nova_hyperv_ci@...`、`openstack@lightbitslabs.com`、`openstack-ci@...`）に対して RF が継続を高確率で誤予測しやすい現象は、RF の重要度構造と整合的である。RF は総レビュー数や活動頻度のような「多いこと」を強い継続シグナルとして学習しやすく、受諾率やトレンドが重要度上位に来ない場合、量が大きいことによる誤った一般化（CI/Bot の大量処理を人間の安定貢献として誤認）が生じやすい。本ケースの IRL のみ正解は、こうした量のバイアスが強い状況で、IRL が相対的に抑制的な判定（離脱）を選びやすいことを示している。

次に、後期（9-12m）で RF が真の継続者を見逃す例が目立つ点も、RF が平均的な応答速度や直近頻度といった「短期の状態量」を重視しやすいことと関係しうる。後期に活動が疎になる、あるいは応答が遅い局面が入ると、RF は上位特徴の変動に強く反応して離脱側に押しやすい。一方で IRL 側は、協力度やコミット量などの「継続に寄与しうる行動指標」や、活動間隔・トレンドの変化を含む複数の要因を同時に考慮するため、局所的な停滞があっても、一定の関与が観測される開発者を過小評価しにくい構造を持つ可能性がある。

以上を踏まえると、本ケーススタディで観測された「IRL のみ正解」は、単に IRL が万能というよりも、RF が重要度上位の少数特徴（平均・総数・頻度）に引き寄せられてしまう場面が存在し、その場面が CI/Bot や活動の質が乏しい高ボリューム開発者、あるいは後期に活動が希薄化する継続者において顕在化する、と解釈できる。実務・研究の含意としては、（i）RF に協力度・コミット量・品質のような行動指標を明示的に導入して「量の罠」を緩和する、（ii）モデルの出力確率を時系列区間ごとに再校正して後期の過小評価を抑える、（iii）IRL の勾配ベース重要度を説明変数として用い、誤りが起きやすいサブタイプ（CI/Bot、低継続率高ボリューム等）を事前に識別する、といった方向が考えられる。これらは、IRL のみ正解ケースを「偶然の当たり」ではなく、モデルが参照している情報の種類の違いから再現可能な現象として位置付ける上で重要である。
